## 2. Folders inside `LearningAPI`

| Folder | What responsibility does it own and why? |
| ------ | ---------------------------------------- |
| `models/` | Defines the database tables as Python classes. Every piece of data (users, cohorts, courses, assessments) needs a structured definition — Django reads these and manages the actual PostgreSQL tables. Split into `people/`, `coursework/`, and `skill/`. |
| `views/` | Handles incoming HTTP requests and returns responses. This is the business logic layer — each file maps to a feature (e.g. `cohort_view.py`, `student_view.py`). |
| `serializers/` | Converts Python model objects to/from JSON. The React frontend speaks JSON; Django speaks Python objects. Serializers translate between the two on every request and response. |
| `tests/` | Automated tests that verify the API behaves correctly. Files like `test_cohort.py` make HTTP requests against the API and assert the responses are correct. |
| `fixtures/` | JSON files full of pre-loaded sample data used to seed the database for development and testing. |
| `migrations/` | Auto-generated files that track database schema changes over time. Run in order to evolve the database structure without losing existing data. |

## 3. What is the Pipfile?

The Pipfile is the dependency manifest for this Python project. It is used by `pipenv`, a tool that manages two things at once: the list of packages the project needs, and an isolated virtual environment to install them into — so they don't conflict with other Python projects on the same machine.

Think of it like a shopping list for Python packages. Instead of manually running `pip install django`, you declare everything in the Pipfile and `pipenv install` sets it all up in one go.

**Key sections:**

- **`[[source]]`** — Where to download packages from (`pypi.org`, the official Python package registry).
- **`[packages]`** — Libraries required to run the app in production (Django, DRF, gunicorn, psycopg2, allauth, etc.).
- **`[dev-packages]`** — Tools only installed on a developer's machine, not in production (pytest, pylint, debugpy).
- **`[requires]`** — Locks the Python version to `3.11.11` so every developer and the server use exactly the same version.
- **`[scripts]`** — Shortcuts, e.g. `pipenv run migrate` runs `python3 manage.py migrate`.

The companion file `Pipfile.lock` (auto-generated) pins the exact version of every installed package so the environment is perfectly reproducible on any machine.

## 4. Key packages

## 5. What does `decorators.py` do?

A decorator is a function that **wraps another function** to add behaviour before or after it runs, without changing the original function's code. The `@` syntax is how Python applies one:

```python
@is_instructor()
def my_view(request):
    ...
```

`decorators.py` defines two access control decorators that protect API views from unauthorised users:

- **`@is_instructor()`** — Checks if the logged-in user belongs to the `Instructors` group. If yes, the view runs normally. If no, it immediately returns a `401 Unauthorized` response — the view never executes.
- **`@is_staff()`** — Same pattern, but checks for the `Staff` group instead.

Without decorators, every protected view would have to repeat the same permission check inline. With this pattern, the check lives in one place and any view can use it with a single line — keeping views focused on their actual logic rather than repeating auth code everywhere.

## 6. What is a serializer, and how does it fit the request/response cycle?

A serializer is a **translator between Python objects and JSON**. Django stores data as Python model instances; the React frontend sends and receives JSON. A serializer is the bridge between the two.

There is no single `serializers.py` — the project uses a `serializers/` folder with one file per model (e.g. `nssuser_serializer.py`, `cohort_serializer.py`).

**Why a Django REST API needs them:**

- **Control what gets exposed** — `fields` whitelists exactly which columns go out in the response, preventing sensitive data from leaking.
- **Validate incoming data** — When the frontend sends a POST request, the serializer checks the data is valid before it touches the database.
- **Django can't send Python objects over HTTP** — A model instance must be converted to a JSON string to travel over the network.

**How a serializer fits the request/response cycle:**

```
REQUEST (JSON in)
   ↓
View receives the request
   ↓
Serializer deserializes JSON → validates it → Python object
   ↓
Model saves it to the database
   ↓
Model returns a Python object
   ↓
Serializer serializes Python object → JSON
   ↓
RESPONSE (JSON out)
```

The view is the coordinator — it calls the serializer in both directions. The serializer never talks to the database directly; that is the model's job.

## 7. One model and what it represents

A Django model is a **Python class that represents a database table**. Each attribute on the class becomes a column in the table, and each instance of the class is one row of data. Django handles all the SQL automatically — you never write raw SQL to create tables or insert rows.

**Example: the `Cohort` model** (`models/people/cohort.py`)

A cohort is a group of students who start NSS together and go through the program as a class. It is one of the most central concepts in the platform — students, courses, and instructors all connect to a cohort.

| Field | Type | Real-world meaning |
| ----- | ---- | ------------------ |
| `name` | `CharField` | The cohort's name, e.g. `"Evening Cohort 10"`. Unique so no two cohorts share a name. |
| `slack_channel` | `CharField` | The Slack channel where this cohort communicates — used to send notifications. |
| `start_date` | `DateField` | When the cohort begins. Used to calculate whether the cohort is currently active. |
| `end_date` | `DateField` | When the cohort graduates. |
| `break_start_date` / `break_end_date` | `DateField` | Tracks scheduled breaks (e.g. holidays) within the cohort's run. |
| `active` | `BooleanField` | Whether this cohort is currently running. Defaults to `False`. |

The model also has a `coaches` property (looks up which staff users are assigned to this cohort) and an `is_active_on_date()` method (checks whether a given date falls within the cohort's run — useful for scheduling and reporting).

## 8. Views vs. viewsets

| Type    | Example | When to use it |
| ------- | ------- | -------------- |
| Plain View | `login_user` / `register_user` in `auth.py` | One-off, custom flows that don't fit standard CRUD — login and registration are unique actions, not typical create/read/update/delete operations. |
| ViewSet | `CohortViewSet` in `cohort_view.py` | Full resources that need standard CRUD operations. DRF automatically routes the HTTP method to the right method (`create`, `retrieve`, `list`, `update`, `destroy`). |

**The key difference:**

A plain view is just a Python function — you manually check `if request.method == 'POST'`, parse the body yourself, and handle one specific action.

A ViewSet is a class where each method maps to an HTTP action. No manual method checking — DRF routes the request automatically. Permissions are declared once on the class rather than repeated per function.

```python
# Plain view — you handle everything manually
@csrf_exempt
def login_user(request):
    if request.method == 'POST':
        # manually parse body, manually return response

# ViewSet — DRF handles routing automatically
class CohortViewSet(ViewSet):
    permission_classes = (CohortPermission,)

    def create(self, request): ...    # POST
    def retrieve(self, request): ...  # GET one
    def list(self, request): ...      # GET all
    def destroy(self, request): ...   # DELETE
```

## 9. What replaces templates and why?

In a traditional Django app, the flow looks like this:

```
Request → View → Model (fetch data) → Template (render HTML) → Response
```

The template took Python data and turned it into an HTML page for the browser.

This project has no templates (except the admin panel). Instead, the **serializer** takes the template's role:

```
Request → ViewSet → Model (fetch data) → Serializer (render JSON) → Response
```

Instead of producing an HTML page, the serializer produces a JSON string.

**Why that makes sense for a REST API:**

A traditional template is tightly coupled to the frontend — it produces HTML that only a browser can display. This project has a separate React frontend (`learn-ops-client`) that runs independently. React doesn't need HTML from the server — it builds its own HTML in the browser. What it needs is raw data, and JSON is the universal format for that.

This separation gives three concrete benefits:

1. **One API, many clients** — the same JSON endpoints could serve the React app, a mobile app, or an external integration without any backend changes.
2. **Frontend and backend are independent** — the React team can change how data is displayed without touching Django, and vice versa.
3. **Cleaner separation of concerns** — Django's only job is to store and serve data correctly. How that data looks on screen is entirely React's responsibility.

In short: **templates produce pages, serializers produce data**. A REST API's job is to serve data, not render UI — so serializers are the right tool.
